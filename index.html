Horas Sindicales

PythonPySide6SQLitePyInstaller

Contexto del problema

La gestión de horas sindicales dependía de hojas dispersas, validaciones manuales y seguimiento reactivo. El resultado era un flujo frágil: inconsistencias en datos, tiempos de cierre largos y baja trazabilidad ante auditorías internas.

Objetivo del sistema

Construir una aplicación desktop que centralizara la operativa diaria, formalizara reglas de negocio y permitiera sincronizar información con hojas compartidas sin comprometer integridad ni mantenibilidad.

Decisiones arquitectónicas clave

  • Modelo por capas con puertos/adaptadores para aislar dominio de UI e integraciones externas.
  • Casos de uso explícitos para separar reglas de negocio de eventos de interfaz.
  • Repositorio SQLite como fuente operativa local con control de esquema versionado.
  • Canal de sincronización desacoplado para Google Sheets, evitando contaminar dominio con detalles de API.

Stack técnico con justificación

  • Python + PySide6: productividad alta para desktop interno y buena ergonomía para flujos administrativos complejos.
  • SQLite: despliegue sin dependencia de infraestructura externa y coste operativo bajo para uso local.
  • gspread/Google Sheets: integración pragmática con herramienta ya adoptada por negocio.
  • PyInstaller + Inno Setup: distribución controlada en Windows con instalación repetible.

Retos técnicos reales

  • Conservar consistencia entre datos locales y remotos bajo edición concurrente.
  • Evolucionar esquema sin romper instalaciones ya desplegadas.
  • Evitar acoplamiento accidental entre componentes UI y reglas de negocio.

Cómo se resolvieron

  • Migraciones SQL versionadas y estrategia explícita de actualización de base.
  • Sincronización encapsulada en casos de uso y adaptadores con contrato claro.
  • Cobertura de pruebas en dominio, aplicación e infraestructura para rutas críticas.

Métricas o impacto

  • Reducción estimada del 35–45% en tiempo de consolidación mensual al eliminar pasos manuales.
  • Disminución estimada >50% de incidencias por incoherencia de datos entre versiones de hoja.
  • Onboarding técnico más corto gracias a arquitectura documentada y release instalable.

Qué mejoraría en versión 2.0

  • Motor de resolución de conflictos con estrategia merge y auditoría por registro.
  • Telemetría de uso y errores para priorización basada en evidencia.
  • Pipeline CI con tests automáticos y empaquetado firmado por release.

Arquitectura

  • Dominio desacoplado
  • Infraestructura aislada
  • Tests automatizados
  • Logging estructurado

ClinicDesk

PythonPySide6SQLitepytest

Contexto del problema

Los procesos de clínica estaban atomizados por tareas y pantallas no coordinadas, lo que generaba fricción en la operación diaria, duplicidades de entrada y dependencia de memoria operativa del equipo.

Objetivo del sistema

Diseñar una aplicación multipágina que consolidara flujo asistencial y administrativo en un producto único, con reglas transversales coherentes y menor coste de mantenimiento funcional.

Decisiones arquitectónicas clave

  • Separación dominio/aplicación/infraestructura/páginas para contener complejidad por capa.
  • Contratos ABC para repositorios y servicios, permitiendo sustituir implementaciones con bajo impacto.
  • Casos de uso como orquestadores de procesos de negocio entre módulos de interfaz.
  • Persistencia SQLite local para garantizar disponibilidad y simplicidad de despliegue.

Stack técnico con justificación

  • PySide6: UI rica, navegación multipágina y control detallado de interacción en escritorio.
  • Python: velocidad de iteración para casos de uso con reglas cambiantes.
  • SQLite: almacenamiento robusto para operación local sin sobrecoste de infraestructura.
  • pytest: validación sistemática de comportamiento en componentes críticos.

Retos técnicos reales

  • Mantener coherencia de reglas entre múltiples pantallas con estados interdependientes.
  • Evitar crecimiento monolítico de lógica acoplada a eventos de UI.
  • Asegurar regresión controlada al introducir nuevas funcionalidades clínicas.

Cómo se resolvieron

  • Extracción progresiva de lógica hacia casos de uso reutilizables.
  • Contratos explícitos entre capa de aplicación e infraestructura para reducir dependencias implícitas.
  • Pruebas de casos de uso y repositorios como red de seguridad para refactor continuo.

Métricas o impacto

  • Reducción estimada del 30% en pasos operativos por consolidación de tareas en una única UI.
  • Menor tasa de retrabajo administrativo por validaciones unificadas entre módulos.
  • Tiempo de entrega de nuevas pantallas más predecible por arquitectura modular.

Qué mejoraría en versión 2.0

  • Instrumentación de eventos de dominio y métricas de rendimiento por flujo.
  • Suite de tests de contrato para adaptadores externos y repositorios.
  • Feature toggles para desplegar cambios funcionales de forma incremental.

Arquitectura

Batería Fotovoltaica

PythonPySide6testsExcel export

Contexto del problema

El análisis de escenarios BESS/FV suele depender de hojas de cálculo difíciles de auditar y mantener, con riesgo alto de errores silenciosos al variar supuestos de demanda, degradación o estrategia financiera.

Objetivo del sistema

Desarrollar un simulador técnico-financiero reproducible, capaz de validar datos de entrada, ejecutar escenarios y exportar resultados trazables para toma de decisiones de ingeniería y negocio.

Decisiones arquitectónicas clave

  • Separación estricta domain/application/infrastructure/ui para proteger reglas de simulación.
  • Casos de uso dedicados para orquestar simulación, persistencia de proyecto y cálculos financieros.
  • Persistencia JSON/Parquet para combinar legibilidad, portabilidad y eficiencia en resultados.
  • Suites de pruebas por nivel (unit, integration, performance) alineadas con riesgo técnico.

Stack técnico con justificación

  • Python 3.11: ecosistema maduro para modelado numérico y validación de datos.
  • PySide6: interfaz guiada para expertos no técnicos sin renunciar a profundidad de parámetros.
  • JSON + Parquet: equilibrio entre interoperabilidad y rendimiento de almacenamiento.
  • pytest: verificación sistemática de exactitud y estabilidad del motor de simulación.

Retos técnicos reales

  • Controlar calidad de datasets heterogéneos provenientes de CSV/Excel.
  • Mantener consistencia entre resultados técnicos y KPIs financieros derivados.
  • Sostener tiempos de respuesta aceptables al aumentar granularidad de simulación.

Cómo se resolvieron

  • Pipeline de validación previa de datos para detectar incoherencias antes del cálculo.
  • Modularización del dominio financiero para aislar supuestos y facilitar recalibración.
  • Pruebas de rendimiento para detectar cuellos de botella en escenarios intensivos.

Métricas o impacto

  • Reducción estimada del 40–60% en tiempo de evaluación de escenarios frente a enfoque manual en hojas.
  • Mayor fiabilidad percibida por separación explícita entre cálculo, persistencia y presentación.
  • Reutilización de simulaciones históricas gracias a almacenamiento estructurado de proyectos.

Qué mejoraría en versión 2.0

  • Ejecución batch de escenarios y comparación automática multiobjetivo.
  • Motor de calibración asistida con sensibilidad y análisis de incertidumbre.
  • Trazas de auditoría por corrida para reporting técnico regulatorio.

Arquitectura

  • Tests automatizados
  • Logging estructurado
